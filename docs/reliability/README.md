## 一、总目标

端侧小龙虾不能继续依赖模型自由发挥来操作手机。新的方向是建设一套“可靠执行内核”：

```text
用户/云端任务
    ↓
任务运行器维护控制流
    ↓
模型只负责生成候选动作
    ↓
动作校验器过滤非法输出
    ↓
步骤约束器防止跳步和提前完成
    ↓
工具执行器操作真实手机
    ↓
成功验收器用证据判断结果
    ↓
失败纠偏、重试、降级、上报
```

核心原则：**模型负责思考，执行内核负责控制、校验、纠偏、验收。**

这套体系的目的不是让端侧模型变大，而是让同样的模型在真实手机环境里更稳：少乱输、少跳步、少误判完成、少无限循环、失败可解释、成功可复现。

## 二、从 forge 项目吸收的关键经验

| forge 经验 | 端侧小龙虾落地方式 | 提升点 |
|---|---|---|
| 控制流和模型上下文分离 | 任务状态、步骤完成情况、重试次数由端侧运行器维护 | 防止模型忘记任务进度 |
| 响应校验器 | 校验模型输出动作格式、动作名、参数、权限 | 减少格式失败和危险动作 |
| 纠偏重试提示 | 输出不合法时，要求模型只返回合法动作 | 减少一次失败就终止 |
| 步骤强制执行 | 必须完成观察、执行、验证等关键步骤 | 防止提前说完成 |
| 错误预算 | 区分模型错误、工具错误、环境错误并设置重试上限 | 防止死循环 |
| 上下文压缩 | 只保留目标、当前页面摘要、最近动作、失败原因 | 降低端侧上下文压力 |
| 代理层思想 | 在现有工具层外增加可靠执行内核，不把逻辑散落到界面层 | 降低架构侵入 |

## 五、最小可落地闭环

第一期不要追求所有应用通吃，先做“微信/消息类任务可靠执行闭环”：

```text
接收任务 → 生成计划 → 观察页面 → 输出结构化动作 → 校验动作 → 执行动作 → 验证结果 → 上报证据
```

第一期验收标准：

- 同一类消息发送任务连续执行十次，成功率不低于八成。
- 每次失败必须有明确失败分类和最后一张截图证据。
- 模型不得直接宣布完成，必须经过成功验收器确认。
- 连续同类失败不得超过配置上限，超过后必须降级或上报云端。
- 所有关键状态变更能从日志还原。

## 七、决策建议

### 方案一：轻量包裹现有任务流

在现有任务流外增加校验、重试、证据层，改动小，能快速见效。

适合第一期。

### 方案二：重构为完整可靠执行内核

新增清晰模块边界，把任务运行、观察、动作、验收、回执全部标准化，长期收益更高。

适合第二期。

### 当前建议

先走方案一，但代码结构按方案二预留包名和接口：

```text
app/src/main/java/io/agents/pokeclaw/reliability/
```

这样第一期能快，第二期不会推倒重来。
